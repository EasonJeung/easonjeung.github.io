Java 注释规范

发表于 2021-10-26 更新于 2021-10-27 分类于 Java

前言

《代码整洁之道》里说：“注释总是一种失败，只是我们无法用代码就清晰表达我们意思的妥协之策。”

当你觉得需要一堆注释来解释你的代码时，你最好去把那堆代码弄干净。

在编程过程中，我们应尽可能让代码含义看起来不言自明。

注释的原则

1. 注释形式统一

整个项目中，保持注释风格、标点、结构等一致。若项目已有规范体系，follow it.

2. 注释与代码的一致性

描述性注释先于代码创建，解释性注释在开发过程中创建，提示性注释在代码完成后创建。

修改代码应同时修改/添加相应的注释，以保证代码与注释的同步。

3. 注释的位置

就近原则：在代码的上方或右侧。

避免在代码行尾添加注释，这样会降低代码可读性。

但在批注变量声明时，右侧行尾注释是被接受的，但注意把行尾注释对齐。

4. 注释的数量

一般占总代码量的 20% 为宜。

5. 删除无用注释

在代码交付或部署前，务必删除无用注释，以免引起日后维护工作的混乱。

6. 必加的注释

典型算法必须添加注释。

代码不清晰或不可移植处必须添加注释。

代码修改处必须添加修改标识的注释。

在循环和逻辑分支组成的代码中添加注释。

为了防止问题反复出现，对错误修复和解决方法的代码使用注释，尤其是在团队环境中。

Java 注释技巧

0. 注释符号与注释之间有空格

//或/*或//*后有一个空格，然后才是注释本身。这样在没有语法高亮时(例如使用txt阅读代码)能够迅速找到注释。

1. 长度限制

无论是代码还是注释，每行建议不要超过120字符，或者80字符。(看个人或团队习惯选择120字符或80字符)

2. 代码尾部注释(应尽量尝试缩短函数)

当代码较长，或是含有多重嵌套时，可以在结尾的 }后注释该闭合的起点，提升代码可读性。不必写得很长，能表示是哪个控制语句范围结束即可。

如果你发现自己想标记右花括号，其实应该做得是缩短函数。 —— 《代码整洁之道》

3.不要加框

给注释加框看起来可能很漂亮，但后期更改注释维护这份”漂亮”可能比你写注释更费工夫。

如下方这种注释，后续更改其中文本时，为了让文字居中且末尾对齐，得很仔细地调整，不要用!!!

4. 注释多行代码

首先选中这多行代码，然后使用键盘快捷键(IDEA是 Ctrl + /)注释多行代码，再按一次则取消注释。

避免使用/* ... code segment ... */，还需要找对应的 /*与*/，没有选中后注释快捷键效率高。

5. 注释作为切换开关，临时测试屏蔽代码片段

例1：此时 code segment_1 未被注释。

只需要删除第一行的第一个 /，code segment_1 被注释。

例2：code segment_1 未被注释而code segment_2 被注释。

只需要删除第一行的第一个 /，code segment_1 被注释而code segment_2 未被注释。

Java 各种注释格式

1. 单行注释 //

a. 单独行注释

注释前最好有一行空行，并与它所注释的代码段拥有相同的缩进层级。

如果一行无法完成，请采用块注释(见下方 2.块注释)。

b. 行头注释

注释该行代码，使其失去意义。

c. 行尾注释

常用来批注变量声明，代码后跟8个(至少4个)空格(或更多空格，个人习惯用 tab )，所有注释必须对齐。

2. 块注释 /* … */

注释若干行，通常用于提供文件、方法、数据结构等的意义与用途的说明，或者算法的描述。

一般位于一个文件或者一个方法的前面，起到引导的作用，也可以根据需要放在合适的位置。

3. 文档注释 /** …*/

注释若干行，并写入javadoc文档。每个文档注释都会被置于注释定界符/** …*/之中，注释文档将用来生成HTML格式的代码报告，所以文档注释必须书写在类、域、构造函数、方法，以及字段(field)定义之前。注释文档由两部分组成——描述与块标记。

3. 接口注释

接口注释采用 /** … */，在满足类注释的基础之上，接口注释应包含接口目的描述、该接口如何被使用以及如何不被使用。

在接口注释清楚的前提下对应的实现类可以不加注释。

4. 构造函数注释

构造函数注释采用 /** … */，需注明构造函数作用。

5. 函数注释

函数注释采用 /** … */，在每个函数或过程前须有必要的注释信息，包括：函数或过程名称；功能描述；输入、输出及返回值说明；调用关系及被调用关系说明等。函数注释里面可以不出现版本号（@version）。

6. 方法注释

方法注释采用 /** … */，对于设置 (Set 方法 ) 与获取 (Get 方法 ) 成员的方法，在成员变量已有说明的情况下，可以不加注释；普通成员方法要求说明完成什么功能，参数含义是什么且返回值什么；另外方法的创建时间必须注释清楚，为将来的维护和阅读提供宝贵线索。

7. 方法内部注释

对复杂的逻辑处理部分，要尽可能给出详细的注释。

控制结构，代码做了什么以及为什么这样做。

8. 全局变量注释

对全局变量的功能、取值范围、哪些函数或者过程存取以及存取时注意事项等的说明。

9. 局部（中间）变量注释

主要变量必须有注释，无特别意义的情况下可以不加注释。

10. 字段(filed)/属性注释

字段描述，属性说明。

11. 常量注释

常量通常具有一定的实际意义，要定义相应说明。
